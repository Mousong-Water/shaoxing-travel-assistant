### Phase 2: 爬虫架构重构 (6月2日 深夜)

**问题识别**: 审查原始爬虫代码，逐条列出21个短板:
- 架构耦合: 请求/解析/存储混在一起
- 稳健性: 无重试、无去重、无频率控制
- 数据质量: 空值不一致、缺少关键字段
- 扩展性: 新增数据源需复制全部代码

**重构方案**:

| 步骤 | 内容 | 方法 |
|------|------|------|
| Step 1 | 抽取公共工具层 | `request_utils.py`(UA池+重试+频控) + `parser_utils.py`(FieldExtractor+正则回退) |
| Step 2 | 抽象爬虫基类 | `BaseScraper` 模板方法模式: `_collect_links()` → `_scrape_details()` → `_save_csv()` |
| Step 3 | 携程爬虫重构 | `CtripScraper(BaseScraper)` 使用 RequestManager + FieldExtractor |

**参考项目/方法**:
- BeautifulSoup CSS选择器多级回退模式 (参考 Scrapy 的 Selector 设计)
- RequestManager 指数退避重试 (参考 HTTP 429 Retry-After 标准)
- 响应码分类处理 (参考 HTTP 规范: 200/403/429/5xx 分别对待)

### Phase 3: 多源爬虫体系 (6月2日-3日)

**设计决策**: 每个爬虫独立、公共逻辑共享、通过 ScraperHub 统一调度

**8大爬虫实现**:

| 爬虫 | 技术方案 | 关键技术点 |
|------|---------|-----------|
| 携程 Ctrip | Playwright+requests混合 | 反检测脚本注入 + Cookie复用 |
| 百度百科 Baike | 静态JSON + requests回退 | 文化背景结构化存储 |
| 大众点评 Dianping | 静态JSON | 18家老字号店铺数据集 |
| 马蜂窝 Mafengwo | 静态JSON | 12条攻略含路线/耗时/贴士 |
| 绍兴政府开放数据 GovAPI | 静态JSON | data.sx.gov.cn 官方API预留 |
| 小红书 Xiaohongshu | Playwright网页版 + 静态 | 免登录访问 xiaohongshu.com/search_result |
| 抖音 Douyin | Playwright优先 + 缓存 + 静态 | 80+关键词搜索 douyin.com/search |
| 知乎/文旅厅/新闻 | 静态JSON | 搜索页公开内容 |

**参考项目**:
- **MediaCrawler** (GitHub 30K+ Stars): 抖音/小红书免登录Playwright方案, 核心思路是注释掉强制登录代码、利用公开搜索页
- **Apify Douyin Scraper**: 商业API设计参考, 关键词搜索 + 结果分页模式

**遇到的关键问题与解决**:
1. ❌ **小红书API全加密** → ✅ 改用Playwright加载网页版搜索页HTML解析
2. ❌ **抖音强制登录弹窗** → ✅ MediaCrawler方案: 跳过登录流程, 公开搜索页可返回结果
3. ❌ **百度百科网络不可达** → ✅ 手工整理15+景点文化背景JSON

### Phase 4: 数据质量体系 (6月3日)

**数据清洗**:
- 地址去噪: 正则删除"讲解服务/卫生间/售票大厅"等噪音信息
- 门票数值化: "免费"→0, "40元"→40
- 时长解析: "1-3小时"→120分钟
- 分类推断: 关键词匹配+名称特征识别

**数据验证**:
- **CrossValidator 多源交叉验证**: 同一景点在多个数据源中出现 → 可信度更高
- **FactChecker 事实检查**: 门票合理性/评分范围/地址一致性
- **4级信任体系**: L4(政府)→L3(3源)→L2(2源)→L1(单源)

**参考方法**:
- 旅游行业数据仓库实体-内容分离模式 (携程/马蜂窝内部架构)
- 维基百科的引用验证机制 (多源交叉验证思路)

### Phase 5: 实体合并引擎 (6月3日)

**核心问题**: 鲁迅故里在7个数据源中各出现1次，共7条记录 → 应合并为1个实体

**EntityMerger 算法**:

```
三重校验合并条件:
  1. 名称相似度 > 0.75 (SequenceMatcher)
  2. 地理距离 < 500m (Haversine公式)
  3. 分类一致 (景点/美食/非遗必须同类型)

字段合并优先级:
  官方 > 马蜂窝 > 大众点评 > 携程 > 小红书 > 百科

评分合并: 多源加权平均
简介合并: 取最长非空值
图片合并: 全部收集去重
```

**关键防护案例**:
- 鲁迅故里 vs 鲁迅外婆家: 名称相似度0.67不高 + 距离10km > 500m → ❌ 不合并
- 鲁迅故里 vs 鲁迅故居: 相似度0.80 + 距离100m + 同分类 → ✅ 合并

**参考方法**:
- 知识图谱实体对齐 (Entity Resolution) 的标准三要素: name similarity + geo proximity + category match
- OpenStreetMap POI去重逻辑

### Phase 7: 住宿推荐引擎 (6月3日)

**评分算法**:

```
综合得分 = 距离得分×40% + 价格得分×30% + 风格匹配×20% + 评分得分×10%

距离得分: Haversine公式 → 酒店到计划景点平均距离 → 归一化
价格得分: 酒店最低价 vs 用户预算(经济300/中等600/高端)
风格匹配: 民宿/高端/温泉/农家乐 vs 用户偏好
家庭模式: ×1.15 加成(亲子设施优先)
```

**参考方法**:
- 携程/美团酒店排序算法 (距离+价格+评分+用户偏好)
- 地理信息系统的 Haversine 球面距离公式

## 三、技术栈总结

| 层级 | 技术 | 用途 |
|------|------|------|
| 爬虫 | requests + BeautifulSoup | 静态页面解析 |
| 爬虫 | Playwright | 动态页面/WAF绕过 (携程/抖音/小红书) |
| 存储 | CSV (utf-8-sig) | 中间交换格式 + 版本可追踪 |
| 存储 | SQLite (WAL模式) | 最终存储 + source_url去重 |
| 合并 | SequenceMatcher + Haversine | 名称相似度 + 地理距离 |
| 验证 | CrossValidator | 多源交叉验证信任评分 |
| 分类 | ContentClassifier | 正则+关键词9大类自动归类 |
| 推荐 | AccommodationScorer | 距离+价格+风格+家庭 加权评分 |

## 四、关键设计决策记录

| 决策 | 选择 | 拒绝的方案 | 原因 |
|------|------|-----------|------|
| 数据库 | SQLite | PostgreSQL | 零配置单文件, 200条数据足够 |
| 前端 | Streamlit(计划) | React+Vite | 聚焦AI算法, 快速原型 |
| 嵌入模型 | BGE-large-zh(计划) | text2vec | 中文语义最优 |
| 向量库 | ChromaDB(计划) | FAISS/Milvus | 零运维, <1000向量足够 |
| 数据合并 | 贪心+三重校验 | 最优搜索(NP-hard) | 可解释性 > 最优性 |
| 爬虫模式 | 静态优先+动态可选 | 全动态实时 | 网络不可达时数据不丢失 |
| 抖音方案 | Playwright公开页 | API逆向/登录爬取 | 合规性 > 数据量 |
| 假数据 | ❌ 禁止生成 | — | 比赛要求真实可信 |
