# Improve Codebase Architecture

参考: mattpocock/skills - improve-codebase-architecture (改编)

## 触发条件

- 每完成一个 Phase 后运行
- 新增 3 个以上模块后运行
- 发现架构异味时 (复制粘贴/循环依赖/配置文件混乱)

## 审查维度

### 1. 冗余检测
```
- 是否存在两份功能相同的代码？
- 是否存在未使用的导入？
- 是否存在从未被调用的函数/类？
- 是否存在已弃用但未删除的文件？
```

### 2. 耦合检测
```
- 模块A 修改后, 模块B 是否必崩？
- 是否存在循环导入？
- 爬虫是否与特定网站结构强耦合 (选择器失效即崩溃)？
```

### 3. 配置漂移
```
- 同一个常量是否在多个文件中定义？
- url/路径/阈值是否分散在各处而非集中管理？
- 两套并行架构是否使用了不同的配置体系？
```

### 4. 数据规范一致性
```
- 字段命名是否统一 (中文/英文混用)？
- 空值处理是否一致 (""/None/0)？
- 来源平台命名是否规范？
```

### 5. 扩展性检查
```
- 新增一个数据源需要改几个文件？
- 新增一个字段需要改几个地方？
- JSON 知识库的字段变更是局部还是全局影响？
```

## 输出格式

```markdown
## 架构审查报告 ({日期})

### 🔴 需要修复
| 问题 | 位置 | 影响 | 修复建议 |
|------|------|------|---------|

### 🟡 建议改进
| 问题 | 位置 | 收益 | 优先级 |
|------|------|------|--------|

### 🟢 当前良好
| 方面 | 说明 |
|------|------|

### 行动计划
1. ...
2. ...
```

## 约束

- 每次审查最多提 5 条改进建议 (避免 overwhelming)
- 明确标注: 立即修复 vs 下一Phase vs 可延后
- 修复后更新 `PROJECT_OVERVIEW.md` 的架构图
