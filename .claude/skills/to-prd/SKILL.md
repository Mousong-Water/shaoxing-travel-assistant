# To PRD

参考: mattpocock/skills - to-prd (改编为学生竞赛版本)

## 触发条件

在开始开发以下任一模块前应先写 PRD:
- 路线规划引擎
- RAG 检索管线
- Streamlit 前端
- 智能体协调器

## PRD 模板

```markdown
# PRD: {模块名称}

## 1. 背景
- 为什么需要这个模块？
- 比赛要求中哪条与此相关？

## 2. 用户故事
- 作为{角色}，我想要{功能}，以便{价值}
- 至少写 3 个用户故事

## 3. 输入/输出
- 输入: 用户偏好 / 景点列表 / 预算 / ...
- 输出: 推荐结果 / 路线方案 / 解说文本 / ...

## 4. 核心算法
- 评分公式
- 排序策略
- 过滤规则

## 5. 依赖数据
- 需要哪些 JSON 字段？
- 需要哪些爬虫数据？
- 是否需要外部 API？

## 6. 前端展示
- Streamlit 页面布局
- 用户交互流程
- 结果展示形式

## 7. 验收标准
- [ ] 输入"xxx"，输出"xxx"
- [ ] 边界情况处理 (无数据/无匹配/极端预算)
- [ ] 响应时间 < 3秒

## 8. 风险
- 数据不足怎么办？
- API 不可用怎么办？
- 算法结果不合理怎么办？
```

## 执行流程

1. 阅读当前讨论上下文，提取已明确的需求
2. 检查 `CLAUDE.md` 确认技术约束
3. 向用户确认 3-5 个关键问题 (不猜测)
4. 填入模板，写入 `docs/prd/{模块名}.md`
5. 确认无误后，触发 `/to-issues` 拆分为任务卡片

## 约束

- 不编造未讨论过的需求
- 不确定的部分标注 `[待确认]`
- PRD 不超过 2 页 A4 纸
