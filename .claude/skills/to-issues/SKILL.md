# To Issues

参考: mattpocock/skills - to-issues (改编为学生团队版本)

## 核心原则: 垂直切片

```
❌ 错误 (水平分层):
  任务1: 写所有数据模型
  任务2: 写所有爬虫
  任务3: 写所有前端页面

✅ 正确 (垂直切片):
  切片1: 住宿推荐 → 数据→评分→前端展示
  切片2: 路线规划 → 数据→算法→前端展示
  切片3: RAG检索 → 数据→嵌入→搜索→前端展示
```

每个切片独立可运行、可演示。

## 执行流程

### Step 1: 分析待办事项

阅读 `PROJECT_OVERVIEW.md` 中"待完成"部分，列出所有未完成模块。

### Step 2: 按垂直切片拆分

每个切片包含:
```
切片名称: {功能描述}
├── 数据准备: 需要哪些 JSON/爬虫数据？
├── 核心逻辑: 算法/评分/检索逻辑在哪实现？
├── 前端展示: Streamlit 页面/组件
└── 集成测试: 端到端验证步骤
```

### Step 3: 分配优先级

按依赖关系排序:
1. 无依赖的模块优先 (如 RAG 检索独立于路线规划)
2. 被依赖的基础模块次之
3. 依赖最多的上层模块最后

### Step 4: 标注工作量

每个切片标注:
- 预估时间 (小时)
- 负责组员
- 依赖的其他切片

## 输出格式

```markdown
## 开发任务板

|---|------|------|------|------|------|--------|------|
| 1 | RAG检索 | JSON→ChromaDB | 混合检索 | 搜索框 | 6h | 组员A | 待开始 |
| 2 | 路线规划 | 景点+距离 | 评分+约束 | 路线卡片 | 8h | 组员B | 待开始 |
| 3 | 多模型 | API Key | 注册+路由 | 对话界面 | 4h | 组员C | 待开始 |
| ... |
```

## 约束

- 不使用专业项目管理工具 (如 Jira/Linear)，团队成员共用此 Markdown 文件
- 每个切片在 `PROJECT_OVERVIEW.md` 中有对应的进度条
